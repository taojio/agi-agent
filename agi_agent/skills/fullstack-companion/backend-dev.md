# 后端开发 Skill

## 任务定位

这是一个 **后端需求分析 + 实现 + 接口文档同步** 的完整开发流程 skill。

目标：
1. 读取产品文档，理解需求
2. 与用户确认需求细节和技术方案
3. 实现后端代码
4. 同步生成 / 更新接口文档，保持接口规范

---

## 第一步：读取产品文档

读取项目根目录下的产品文档（默认路径 `产品文档.md`）。

如果文件不存在，询问用户产品文档的位置。

同时读取以下文件（如存在）：
- `接口文档.md` — 已有的接口文档
- `后端实现文档.md` — 已有的后端实现文档

如果项目有 `CLAUDE.md` 或其他代码规范文档，也一并读取以了解项目约定。

---

## 第二步：需求分析与确认

根据产品文档，输出以下内容并等待用户确认：

### 2.1 需求理解

- 列出本次需要实现的功能点
- 标注每个功能点的优先级（核心 / 次要 / 可选）
- 标注不确定或有歧义的需求，列为「待确认项」

### 2.2 技术方案

- 数据模型设计（表结构、字段、关联关系）
- API 接口设计（路径、请求参数、响应格式）
- 核心业务逻辑说明
- 涉及的文件清单

### 2.3 待确认项

将所有不确定的需求和技术细节汇总，逐条列出，等待用户逐一确认。

**重要：必须等待用户确认后再进入实现阶段。**

---

## 第三步：代码实现

用户确认方案后，按照项目已有的代码风格和目录结构进行实现。

### 实现原则

- **遵循项目惯例** — 先阅读项目中已有的代码，理解并遵循已有的代码风格、目录结构、命名规范
- **职责分离** — Controller / Handler 负责参数校验和路由分发，Service / Logic 层承载核心业务逻辑
- **统一响应格式** — 使用项目已有的统一响应格式，如果没有则与用户确认
- **参数校验** — 使用框架提供的校验机制，在入口层校验
- **最小改动原则** — 只实现需求要求的功能，不做额外的优化或重构

### 实现顺序（按项目实际情况调整）

1. 数据层 — 数据库迁移 / 表结构 / 数据模型
2. 业务层 — Service / 核心业务逻辑
3. 接口层 — Controller / Handler / 路由注册
4. 其他 — 配置、中间件等（如需要）

---

## 第四步：同步接口文档

代码实现完成后，**必须** 同步更新 `接口文档.md`：

### 接口文档规范

每个接口必须包含以下内容：

1. **接口路径** — `METHOD /api/xxx/yyy`
2. **请求参数** — 表格形式（参数名、类型、是否必填、说明）
3. **请求示例** — JSON 格式
4. **响应示例** — JSON 格式（成功 + 失败）
5. **响应字段说明** — 表格形式（字段名、类型、说明）
6. **副作用说明** — 如果接口有级联操作（如删除时清理关联数据），必须说明

### 文档同步规则

- 新增接口 → 在文档对应章节追加
- 修改接口 → 更新文档中对应接口的参数 / 响应 / 说明
- 删除接口 → 从文档中移除
- 保持文档结构与代码实现一致
- 接口按模块分组
- 保持请求 / 响应示例与实际代码逻辑一致

---

## 第五步：输出总结

完成后输出：

### 5.1 已实现功能

列出本次实现的功能点及对应文件。

### 5.2 接口变更

列出本次新增 / 修改 / 删除的接口。

### 5.3 数据库变更

列出本次新增 / 修改的表和字段。

### 5.4 待处理项

列出未完成或需要后续处理的内容。

---

## 迭代开发

当用户提出新的需求或修改时，重复以上流程：

1. 重新读取产品文档（可能已更新）
2. 分析增量需求
3. 与用户确认方案
4. 实现代码
5. **同步更新接口文档** — 这一步不可省略

每次修改接口后都必须确保接口文档与代码保持同步。
