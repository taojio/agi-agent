# Fullstack Companion — 全栈开发伴侣

一个 Claude Code Skill，让 AI 像一位经验丰富的全栈工程师一样陪你从需求到上线。

---

## 它解决什么问题？

日常开发中，前后端协作经常面临这些痛点：

- **接口文档和代码不同步** — 后端改了接口忘了更新文档，前端按旧文档写完才发现对不上
- **前后端各自为战** — 没有统一的开发节奏，联调时才暴露大量问题
- **重复沟通成本高** — 每次需求变更，前后端要反复确认字段、格式、参数
- **代码规范难以落地** — 团队有规范但没人持续执行，时间一长就走样

Fullstack Companion 把这些问题变成了自动化的流程约束。

---

## 核心优点

### 1. 一键启动，零配置

```
/fullstack-companion
```

不需要告诉它你用的是什么框架。Skill 会自动扫描项目特征文件（`composer.json`、`package.json`、`go.mod`、`pom.xml` 等），识别出项目类型和技术栈，然后引导你进入正确的开发流程。

支持的技术栈：
- **后端**：Laravel、Go/Gin、Django、FastAPI、Flask、Express、NestJS、Spring Boot、Rust
- **前端**：Vue、React、Next.js，以及通用 Vite/Webpack 项目

### 2. 前后端流程通过「接口文档」打通

这是本 skill 最核心的设计：**接口文档是前后端的唯一契约**。

```
产品文档 ──→ 后端开发 ──→ 接口文档 ──→ 前端开发
                           ↑
                     自动生成 & 同步
```

- **后端流程**：每次实现代码后，强制同步更新 `接口文档.md`，保证文档和代码一致
- **前端流程**：每次收到新需求时，强制重新读取接口文档，获取后端最新的接口变更
- **全栈模式**：先跑后端流程产出接口文档，再跑前端流程消费接口文档，天然串联

这意味着：
- 后端不可能「改了接口不更新文档」 — 流程不允许跳过
- 前端不可能「按过期文档开发」 — 每次都强制刷新
- 联调时不会出现「字段对不上」 — 双方基于同一份文档

### 3. 先确认再动手，不浪费时间

无论前端还是后端，在写代码之前都有一个**需求确认环节**：

- 后端：列出功能点、技术方案、数据模型、待确认项，等用户确认后才开始实现
- 前端：列出页面结构、接口对齐结果、待确认项，确认后才生成代码

好处是：
- 避免「做完才发现需求理解错了」的返工
- 所有歧义和不确定项在开始前暴露，不藏在代码里
- 用户对整体方案有掌控感，不是被动等结果

### 4. 接口文档规范自动执行

后端每个接口自动按统一格式输出：

| 规范项 | 内容 |
|--------|------|
| 接口路径 | `METHOD /api/xxx/yyy` |
| 请求参数 | 表格（名称、类型、必填、说明） |
| 请求示例 | JSON |
| 响应示例 | JSON（成功 + 失败） |
| 响应字段 | 表格（名称、类型、说明） |
| 副作用 | 级联操作说明 |

不需要人工维护格式，每次接口变更自动保持一致。

### 5. 前端代码规范内置，可插拔

内置一套完整的 Vue3 前端代码规范（`code-style.md`），包括：

- **文件拆分规范** — 禁止单文件堆砌，强制拆为 `index.vue` + `hook.tsx` + `types.ts` + `enums.ts`
- **完整代码模板** — 列表页、详情页、弹窗页、API 封装、Mock 数据的标准模板
- **关键规则** — reactive 类型定义、列定义放 hook、枚举放 enums、三态覆盖等

这套规范是**可插拔**的：
- 默认使用内置规范
- 可以指定其他规范 skill 替换
- 可以选择不使用规范约束
- 项目已有 `CLAUDE.md` 等规范文档时，自动以项目规范为准

### 6. 框架无关的通用设计

后端开发流程**不绑定任何特定框架**：

- 不预设 Controller/Service 的命名方式
- 不预设数据库迁移的写法
- 不预设响应格式

而是要求 AI 先阅读项目已有代码，理解并遵循项目的既有风格。这意味着同一个 skill 可以在 Laravel 项目、Go 项目、Spring Boot 项目中通用，无需修改。

### 7. 支持迭代开发，不是一次性工具

每个子流程都设计了迭代循环：

- **后端**：需求变更 → 重读产品文档 → 确认增量方案 → 实现 → 同步文档
- **前端**：需求变更 → 重读接口文档 → 检测变更 → 确认方案 → 实现

前端还内置了**接口文档变更检测**：
- 新增接口 → 提醒用户
- 参数变更 → 检查是否影响已有页面
- 接口删除 → 检查是否破坏已有功能

## 目录结构

```
.claude/skills/fullstack-companion/
├── SKILL.md            # 入口 — 项目检测 & 流程路由
├── backend-dev.md      # 后端开发流程
├── frontend-dev.md     # 前端开发流程
└── code-style.md       # 前端代码规范（Vue3 模板 + 文件拆分规范）
```

所有子流程作为入口 skill 的内部资源，对外只暴露一个命令 `/fullstack-companion`。

---

## 使用方式

### 快速开始

在 Claude Code 中输入：

```
/fullstack-companion
```

Skill 会自动检测项目类型并引导你进入对应流程。

### 手动指定模式

如果自动检测不准确，可以在确认环节输入 `frontend` 或 `backend` 手动切换。

### 全栈开发

在全栈项目中选择「全栈开发」模式，会按以下顺序执行：

1. 后端流程 → 产出代码 + 接口文档
2. 前端流程 → 读取接口文档 + 实现页面

## 适用场景

| 场景 | 效果 |
|------|------|
| 新功能开发 | 从产品文档到前后端代码一步到位 |
| 接口变更 | 后端改接口自动更新文档，前端自动感知变更 |
| 前后端分离协作 | 接口文档作为契约，双方独立开发不跑偏 |
| 代码规范落地 | 规范内置在流程中，不依赖人工 review |
| 跨项目前端开发 | 支持从其他项目路径读取接口文档 |

---

## 设计理念

> **文档不是开发的副产品，而是开发流程的核心节点。**

传统开发中，文档往往在代码写完后补，容易过期。Fullstack Companion 把文档写入流程的关键路径上 — 后端不更新文档就无法完成流程，前端不读文档就无法开始开发。这让文档从「应该做但经常忘」变成了「不做就走不下去」。
