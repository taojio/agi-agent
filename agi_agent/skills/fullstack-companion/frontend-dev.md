# 前端开发 Skill

## 任务定位

这是一个 **基于接口文档的前端页面开发** skill。

目标：
1. 读取后端接口文档，理解数据结构和接口约定
2. 根据用户需求实现前端页面
3. 遵循代码规范（内置默认规范或用户指定的规范 skill）

---

## 第零步：确认接口文档位置

接口文档可能不在当前项目中。启动时询问用户：

> 请确认接口文档的位置（默认：当前项目根目录下的 `接口文档.md`）。
> 如果接口文档在其他项目中，请提供完整路径。

用户确认后记录路径，后续步骤中使用该路径读取文档。

---

## 第一步：读取接口文档

读取用户确认的接口文档路径，解析以下信息：

- 所有接口的路径、方法、参数、响应格式
- 数据模型和字段定义
- 接口间的关联关系
- 统一响应格式

**重要：每次收到用户新的输入或需求变更时，必须重新读取接口文档，因为后端可能已经更新了文档。**

---

## 第二步：确认代码规范

询问用户是否需要使用代码规范约束：

> 是否需要使用代码规范 skill？
> 1. 使用默认代码规范（读取本 skill 同目录下的 `code-style.md`）
> 2. 指定其他代码规范 skill（请输入 skill 名称或路径）
> 3. 不使用代码规范约束

如果用户选择使用代码规范，读取对应文件并在后续开发中遵循其约束。

默认代码规范资源位于本 skill 同目录下的 `code-style.md`。

同时，如果项目中存在以下文件，也一并读取作为规范参考：
- `CLAUDE.md`
- `docs/frontend-patterns.md`
- `docs/component-catalog.md`
- `docs/api-conventions.md`

---

## 第三步：需求分析

根据用户描述的需求，结合接口文档，输出以下内容：

### 3.1 页面结构

- 需要创建 / 修改的页面和组件
- 页面布局和交互流程
- 需要调用的接口列表

### 3.2 接口对齐

- 列出本次需求涉及的接口
- 确认接口参数和响应字段是否满足页面需求
- 如果接口不满足，标注「需要后端补充」并告知用户

### 3.3 待确认项

- 交互细节不明确的地方
- 接口字段与页面需求不匹配的地方
- 其他需要用户确认的内容

---

## 第四步：代码实现

用户确认后，按照以下原则实现：

### 实现原则

- **遵循项目惯例** — 先阅读项目中已有的页面代码，理解并遵循已有的代码风格、目录结构、组件使用方式
- **优先复用** — 优先使用项目已有的组件、hooks、工具函数，避免重复造轮子
- **接口对齐** — 请求参数和响应处理严格按照接口文档实现，不擅自修改字段名或数据结构
- **类型安全** — 为接口请求和响应定义类型（如项目使用 TypeScript）
- **最小改动** — 只实现需求要求的功能，不做额外优化或重构

### 实现顺序（按项目实际情况调整）

1. 类型定义 — 接口请求 / 响应类型
2. API 封装 — 接口调用函数
3. 页面组件 — 主页面结构
4. 子组件 — 弹窗、表单、列表等
5. 状态管理 — 如需要

---

## 第五步：输出总结

完成后输出：

### 5.1 已实现功能

列出本次实现的功能点及对应文件。

### 5.2 调用的接口

列出本次使用的接口及用途。

### 5.3 未实现内容

列出未完成或需要后续处理的功能。

### 5.4 接口问题

如果发现接口文档与需求不匹配的地方，列出并建议后端调整。

---

## 迭代开发

当用户提出新的需求或修改时：

1. **重新读取接口文档** — 后端可能已更新接口，必须读取最新版本
2. 分析增量需求
3. 与用户确认方案
4. 实现代码
5. 如果发现接口变更影响已有代码，主动提醒用户

---

## 接口文档变更检测

每次重新读取接口文档时，与上次读取的内容对比：

- 如果有新增接口 → 提醒用户是否需要使用
- 如果有接口参数变更 → 检查是否影响已实现的页面，如有影响则提醒并修复
- 如果有接口删除 → 检查是否影响已实现的功能，如有影响则提醒用户
